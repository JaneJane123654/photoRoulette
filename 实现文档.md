1. 功能概述

本项目是一个基于本地相册的“盲盒式”图片浏览应用。核心玩法为：

绝对随机： 每次启动以完全随机的顺序展示本地相册图片。

手势交互： 采用类似 Tinder 的卡片层叠 UI，支持左滑/右滑（跳过当前图片），上划（触发删除本地图片操作）。

开关控制： 提供“上划删除功能”的全局配置开关。

无缝恢复： 当用户上划删除但拒绝了系统删除权限时，卡片需带有物理弹簧动效优雅回退到屏幕中央。

1. 关键技术选型 (现代化安卓全家桶)

开发语言： Kotlin (纯 Kotlin 项目)

UI 界面框架： Jetpack Compose (手搓卡片层叠与物理弹簧手势)

图片加载： Coil (协程图片加载，负责自动降采样和三级缓存)

架构模式： MVVM (Model-View-ViewModel) + Clean Architecture 原则

异步流控： Kotlin Coroutines & StateFlow (保障主线程绝对流畅)

本地配置存储： Jetpack Preferences DataStore

图片数据源： MediaStore API (仅读取 ID，拒绝直接读流)

模块三：业务控制 —— ViewModel 与撤回机制 (MainViewModel)

开发指导：

维护一个 StateFlow<List<Long>> 作为当前的卡片队列（通常只需向 UI 暴露前 3 张即可，不要把几万张暴露给 UI 层）。

待定区设计（Pending State）：
当触发“上划删除”时，从队列移除该 ID，但将其存入 var pendingDeleteId: Long?。

撤回逻辑（Rewind）：
如果系统删除回调返回 RESULT_CANCELED，将 pendingDeleteId 重新插回 UI 队列的头部，UI 层的 Compose 监听到数据变化，配合 AnimatedVisibility 或重置 Animatable，播放卡片飞回动画。

7. 性能优化与防 OOM 策略 (黄金法则)

绝对懒加载限制： 无论用户手机里有 5 万还是 10 万张照片，UI 树中永远只存在 3 个 Card 组件（当前张、下一张、下下一张）。利用 Box 配合 zIndex 实现。

Coil 的自动降采样： 在传入 Coil 的 AsyncImage 时，由于 Compose 组件有明确的尺寸修饰符（如 Modifier.fillMaxSize().padding(16.dp)），Coil 会在底层自动计算合适的 TargetSize，从磁盘读取时自动降采样，10MB 的图加载进内存可能只有 500KB。

生命周期感知： ViewModel 中的 Flow 必须使用 collectAsStateWithLifecycle() 收集，确保应用切到后台时，动画和图片加载任务自动挂起，不耗电。

1. 项目交付标准 (DoD)

流畅度： 在中端机型上，连续快速滑动卡片无掉帧，卡片跟随手指 0 延迟，释放后的物理弹簧动画自然（无僵硬的线性动画）。

内存指标： 首次加载 10,000 张图库数据耗时 < 500ms，滑动过程中 App 内存占用峰值稳定在 100MB-150MB 之间，无内存泄漏。

边界测试通过：

上划删除触发系统弹窗时，App 切后台再切回来，状态不丢失。

用户在系统弹窗中点击“取消”，卡片必须以动画形式飞回，数据列表不能发生错位。

针对 Android 14 手机，选择“仅部分授权”时，App 不崩溃且能正确洗牌这部分照片。

注意以下几个改善点：
可以改进或需要补充的 6 个关键点：

1. 架构与稳定性隐患：系统弹窗导致“进程被杀（Process Death）”
当前方案的盲区： 当用户上划触发删除，App 调用 MediaStore.createDeleteRequest 呼出系统级的确认弹窗时，你的 App 实际上会被推到后台（onPause / onStop）。在低内存手机上，系统为了给这个弹窗和底层的媒体服务腾出内存，极有可能把你的 App 进程直接杀掉。
后果： 当用户点击“取消”或“允许”后，系统重新拉起你的 App。此时 ViewModel 会被重建，你存在内存里的 pendingDeleteId 和当前的 卡片游标（Index） 全部丢失。用户不仅看不到优雅的“卡片回退动画”，反而会发现盲盒被重置到了第一张。
改进方案：
必须引入 SavedStateHandle。在 ViewModel 中，将 pendingDeleteId 和当前的 currentIndex 存入 SavedStateHandle。这样即使进程被系统意外回收，重新拉起时也能精准恢复刚才的状态，并无缝衔接动画。
2. 数据源同步隐患：MediaStore 的“幽灵数据”
当前方案的盲区： 应用只在启动时查询一次 MediaStore 并洗牌。但如果用户中途切到后台，用系统相机拍了新照片，或者在相册里删了照片，再切回你的 App，你的 ID 列表就成了“脏数据”。
后果： 滑到被系统相册删掉的照片时，Coil 会加载失败，卡片变成空白或黑块。
改进方案：
需要注册一个 ContentObserver 或使用 Kotlin Flow 监听 MediaStore 的变化。
当检测到外部数据变化时，不需要全盘重新洗牌（会打断用户当前体验），而是静默过滤掉列表中已经失效的 ID，并在队尾补充新插入的 ID。
Coil 容错处理： 在 UI 层给 Coil 的 AsyncImage 配置 error 占位图。更进阶的做法是：监听 Coil 的加载状态，如果发现该 ID 的图片已损坏或不存在，在代码层面自动触发“跳过”动画，不要让用户对着一张坏图发呆。
3. Compose 性能瓶颈：手势动画的“重组（Recomposition）灾难”
当前方案的盲区： 文档提到了使用 Animatable 的 offsetX 和 offsetY，如果在 Compose 中直接把这些 offset 写在 Modifier.offset 或外层布局上，手指滑动的每一帧（16ms内）都会触发 Compose 的重新布局（Layout）甚至重组（Recomposition），导致严重掉帧。
改进方案：
必须在文档的“性能优化”章节强调：所有的位移、旋转、缩放、透明度变化，必须强制写在 Modifier.graphicsLayer { ... } 内部。
graphicsLayer 是 Compose 的性能救星，它允许在 GPU 层面直接做矩阵变换，绕过 Compose 树的重组和重绘，这样就算是在低端千元机上，手势拖拽也能满帧（60fps/120fps）运行。
4. 预加载策略的局限：滑动过快时的白屏
当前方案的盲区： 内存中保留 3 张卡片（当前、下1、下2）可以防 OOM，但如果用户疯狂连刷（比如 1 秒内左右狂滑 5 次），UI 组件根本来不及生成，Coil 也来不及从磁盘解码，会出现明显的白屏等待。
改进方案（双重预加载）：
UI 树里保留 3 个节点没错，但我们要脱离 UI 提前让 Coil 干活。
在 ViewModel 层面，获取当前 index 往后的 5-10 个 ID，直接调用 Coil 的底层 API ImageLoader.enqueue(ImageRequest) 进行后台预加载（仅解码放入内存缓存，不渲染）。
这样当用户疯狂滑动时，图片早就以 Bitmap 的形式躺在 Coil 的内存缓存里了，UI 顶上来的瞬间 0 耗时展示。
5. 产品体验（UX）：纯随机的“伪随机灾难”（Spotify 痛点）
当前方案的盲区： 简单的 Collections.shuffle() 会产生真正的随机。但在人类的感知里，“真随机”往往意味着“不随机”。（比如连续抽到 5 张同一天拍的相似照片，用户会觉得你算法坏了）。
改进方案（算法优化）：
引入**“打散算法（Smart Shuffle）”**。在拿到所有图片数据（带有拍摄时间戳）后，先按时间/月份分组，然后从不同时间跨度的桶里交替抽取图片组成新的列表。
这样用户上一张看到的是 5 年前的大学毕业照，下一张可能是昨天的晚餐，视觉冲击力和盲盒的“惊喜感”会成倍提升，这也是做这个产品最核心的价值。
6. 上划删除的容错体验
当前方案的盲区： 上划（Up-swipe）这个动作在单手握持手机时其实很容易误触。虽然有系统弹窗拦截，但频繁误触会导致频繁弹窗，极其打断心流。
改进方案：
提供“撤销（Undo）”安全网：除了系统弹窗，也可以在 App 内部做一个类似于 Gmail 的底部 Snackbar 提示：“已放入回收站[撤销]”。
手势阈值区分： 左右滑动的触发阈值可以设小一点（比如滑动屏幕宽度的 20%），但上划删除的触发阈值必须设大（比如上划距离超过屏幕高度的 40% 且速度要快），以极大地降低误触率。
